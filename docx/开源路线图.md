# pizz 开源路线图

> 版本：0.1.0 | 更新日期：2026-05-09

---

## 一、总体策略

**路线：先闭源打磨 → 种子用户验证 → 企业版上线 → 开源发布**

核心原则：
- **开源版是完整产品**，不是功能残缺的诱饵
- **功能分层，不是基础设施分层** —— 不锁登录，锁团队协作
- **个人开发者用开源版完全够用**，只有团队/公司才需要付费

## 四、开源版 vs 企业版功能边界

```

```

## 五、盈利模型

- **企业版订阅：** ¥499/年/席位（主收入来源）

- **技能市场抽成：** 用户发布付费技能包，平台抽 15-30%
- **API 代理服务：** 统一 API 网关，赚取批量折扣差价

## 六、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 巨头入场（Cursor/Copilot 免费） | 用户流失 | 差异化：多供应商 + 双角色 + 国内供应商原生支持 |
| 开源后被人 fork 竞争 | 社区分裂 | AGPL 许可证（防止云服务商白嫖）+ 运营官方社区 |
| 企业版付费意愿低 | 收入不足 | 降低起售价、增加技能市场收入 |
| 安全漏洞 | 声誉风险 | Phase 1 完成安全基础、建立漏洞披露流程 |
| 单人开发瓶颈 | 进度慢 | 开源后吸引贡献者、考虑融资 |

## 七、许可证选择（待定）

| 许可证 | 优势 | 劣势 |
|--------|------|------|
| **Apache 2.0** | 最宽松，商业友好，社区接受度高 | 允许云服务商白嫖不贡献代码 |
| **AGPL-3.0** | 防止云服务商白嫖，修改必须开源 | 企业用户担忧合规，降低采用率 |
| **BSL** | 商业源代码许可证，自动变为开源 | 复杂度高，社区认知度低 |

**建议：** 核心代码用 **Apache 2.0**，企业版插件用专有许可证。技能市场作为独立社区仓库发布。
